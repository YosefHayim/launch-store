アプリを世に出すことは、単なるビルドにとどまりません。署名のセットアップ、[App Store Connect](https://developer.apple.com/app-store-connect/) / [Play Console](https://play.google.com/console) の設定、アプリ内課金、ストア掲載のメタデータ、アップロード、そしてその後の OTA（無線越し）アップデート。EAS はビルドとサブミットを担いますが、それ以外は Apple と Google のポータルや一握りのツールに散らばっています。Launch は **リリース全体** を一つのローカルで宣言的なワークフローに集約します。署名をプロビジョニングし、ストアの商品を整合させ、ネイティブプロジェクトを生成し、バイナリをビルド・署名し、デバイスごとの実際のダウンロードサイズを報告し、成果物を保存し、テスト用トラックへアップロードします。すべてあなた自身のハードウェア上で、鍵はローカルのキーチェーンに留めたまま行われます。iOS の署名には Mac が必要です。Mac がない場合、Launch は **あなた自身の** AWS アカウント内のクラウド Mac でビルドするか、Expo EAS に処理を引き継ぎます — [Mac なしでビルドする](#mac-なしでビルドする) を参照してください。

> **初めての方へ** `launch demo` を実行すると、パイプライン全体を 60 秒でシミュレーションした体験ができます — セットアップもビルドもアカウントも不要です。`launch` を初めて実行したときにも自動再生されます。

## Launch を選ぶ理由

ビルドそのものが難しかったことは一度もありません — 難しいのはその周囲のリリース作業です。Launch はその領域全体を、ローカルかつオープンソースで引き受けます。

- **リリース全体を、一つのワークフローで。** 署名、ストアの商品、ビルド、サイズチェック、アップロード、そして OTA アップデートが、一つの `launch.config.ts` とわずかなコマンドから生まれます — 十数個のダッシュボードや CI スニペットは要りません。
- **コードとしてのストア設定。** `launch sync` は、一つの型付き `launch.config.ts` から、アプリ内課金・サブスクリプション・価格・ケイパビリティを App Store Connect に整合させます。さらに十数個のコマンドが、Game Center、Wallet、App Clips、アプリ内イベント、A/B 実験、地域、そして Google Play カタログ全体をカバーします — 加えて掲載情報には `launch metadata` を使います。これらは EAS では手作業でのクリックに委ねられている部分です。
- **コンピュート費用は $0、ビルドは無制限。** EAS はビルドごとに課金し、無料枠を 45 分のタイムアウトで制限し、有料プランでは **月額 $19〜$199** に加えて超過料金がかかります。Launch はあなた自身のマシンでビルドします — メーターも、キューも、タイムアウトもありません。
- **鍵はローカルに留まる。** 配布証明書、App Store Connect API キー、Android アップロードキーは OS キーチェーンに留まります。Launch が Apple に送るのは CSR だけです。（Mac なしでのビルドは唯一の例外です — 下記参照。）
- **ロックインは一切なし。** MIT ライセンスで、`fastlane`、Gradle、各プラットフォーム自身の API の上に構築されており、ストレージ／認証情報／ビルド／サブミットのプロバイダーを差し替え可能です。あとで移行しなければならない独自仕様は何もありません。
- **動かしながら教えてくれる。** どのコマンドにも `--explain` を付ける — あるいは `launch demo` を実行する — だけで、各ステップ（CSR、プロビジョニングプロファイル、TestFlight、Play トラック、サブスクリプショングループ）が平易な言葉で展開されます。

## Launch とは何か — そして何でないか

**Launch とは** エンドツーエンドのリリースツールです。ソースからストアまでの道のり全体 — ビルド、コード署名、サイズチェック、コードとしてのストア設定、アップロード、一般公開リリース、そして OTA アップデート — を、iOS と Android の双方について、あなた自身のハードウェア上で引き受けます。

**Launch とは違う** のは、単なる App Store Connect の SDK や ASC の MCP サーバーではないという点です。それらは Apple API の一部分をラップするにすぎません。Launch は Apple **と** Google にまたがるリリース全体を、API ラッパーが触れない署名・ビルド・OTA アップデートを含めて駆動します。単なる API クライアントではなくセルフホスト型の Expo EAS が欲しいなら、それが Launch です。

## 機能

**セットアップと検証**

- **設定はワンステップ。** `launch init` はあなたのアプリ — `apps/*` モノレポを含む — を検出し、コメント付きの `launch.config.ts`（および出発点となる `.env.example`）を書き出します。書き出すのは設定だけで、認証情報やネイティブプロジェクトには一切触れません。
- **鍵はキーチェーンに。** `launch creds set-key` は `~/Downloads` 内の `AuthKey_*.p8` を見つけ、Apple に対して検証し、OS のシークレットストアに格納します。`creds setup` はアプリ ID を登録し、署名用アセットを作成または再利用します。マルチアカウント対応の `creds use/rename/remove` でチーム間を切り替えられます。`creds push-key` は一度しかダウンロードできない APNs 認証キー（Apple には作成用 API がありません）を保管庫に収め、必要に応じて再エクスポートします。
- **平文の `.env` ではなくシークレットを。** `launch secret set <NAME>` はビルド用のシークレットを OS キーチェーン（アプリ／プロファイル単位でスコープされる）に格納し、ビルド環境に注入します — これにより、本物のシークレットがコミットされた `.env` の外に保たれます。`secret list`（マスク表示）と `secret rm` がこれを補完します。
- **`launch doctor --fix`。** iOS/Android のツールチェーンを検出し、brew でインストール可能な不足ツールを一度の同意でインストールし（`--yes` で CI／エージェント向けにスキップ）、ストア側の障害 — App Store Connect レコードの欠如、署名されていない Apple の契約 — を指摘し、既知のネイティブ設定の落とし穴（不正なバンドル ID／パッケージ、`backgroundColor` のないスプラッシュ）について Expo の設定を検証します。これは `launch build` が前段で実行するのと同じプレフライトであり、一行の設定ミスがビルドではなく一瞬で失敗するようになります。さらに、EAS が黙って見過ごす二つのサブミット時の落とし穴も表面化させます。すなわち、**輸出コンプライアンス** の回答（`ios.config.usesNonExemptEncryption` から一度読み取り、`--fix` を付ければ最新ビルドに対して API 経由で回答されます）と、一度きりの手動 **App プライバシー** アンケートです。後者については Apple が API を一切公開していないため、Launch は正確なチェックリストを印字し、サブミットを不意にブロックさせないようにします。
- **署名ステータスを一目で。** `launch setup ios` は iOS のプロビジョニングを端から端まで報告します — アクティブなアカウント、App ID、ケイパビリティ、配布証明書、プロファイル、登録済みデバイス — そして `--provision` を付けると、`launch creds setup` と同じく証明書と App Store プロファイルを確実に用意します。

**App Store Connect を設定 — コードとして**

以下の各セクションは `launch.config.ts`（またはそのスタンドアロンな `*.config.json` サイドカー）に宣言され、読み取り専用の **プラン → あなたの確認 → 適用** で整合されます — べき等に、ライブ版や審査中のバージョンには決して触れません。これは EAS が App Store Connect のウェブサイトに丸ごと委ねている領域です。

- **商品、価格、掲載情報。** `launch sync` は、アプリ内課金、サブスクリプション、ケイパビリティ、そして **価格** を App Store Connect に整合させます — そして同じパスで、ロケールごとの **掲載文言**（名称、サブタイトル、説明、キーワード、最新情報、プライバシー／サポート／マーケティング URL）、**スクリーンショット**、**アプリプレビュー** も、すべてのアプリについて一度に整合させます。
- **サブスクリプションのオファー。** `launch offers` はオファーコードと、プロモーション／導入／ウィンバックのオファー、加えてプロモーション購入の並び順を整合させます。`launch offers generate-codes/list/deactivate` は CLI からオファーコードのキャンペーンを操作します。
- **リリース属性。** `launch release-config` は App Store の **年齢レーティング、カテゴリ、基本価格、App Review 詳細**（連絡先＋デモアカウント）を編集可能なバージョンに整合させます。
- **アプリのアイデンティティとエンタイトルメント。** `launch game-center`（実績とリーダーボード）、`launch wallet`（Apple Pay マーチャント ID と Wallet パスタイプ ID）、`launch app-clips`（App Clip カードのアクション＋サブタイトル）、そして `launch eu-distribution`（DMA 向けの EU 代替配信ドメイン＋パッケージ署名鍵） — `spaceship` が公開しているものの EAS が扱わない、ポータルでクリックして行うチームセットアップです。
- **マーチャンダイジングとプレゼンス。** `launch availability`（アプリを販売する地域）、`launch custom-pages`（代替プロダクトページ）、`launch experiments`（プロダクトページの A/B テスト）、`launch accessibility`（アクセシビリティの栄養成分表示ラベル）、`launch events`（アプリ内イベント）、そして `launch metadata pull/push`（**iOS _と_ Android** 双方の掲載情報全体 — `eas metadata` は iOS のみ）。

**Google Play を設定 — コードとして**

- **Play の商品とサブスクリプション。** `launch play-products` と `launch play-subscriptions` は、App Store Connect を駆動するのと **同じ** `launch.config.ts` カタログから、Play のアプリ内商品とサブスクリプション（基本プラン＋オファー）を整合させます — 一つの信頼できる情報源で、両方のストアを。
- **トラックとレビュー。** `launch play-tracks` はトラックの状況を表示し、選んだ展開率でリリースノートとともにビルドをトラックへ **プロモート** します（テスターグループの読み取り／設定も可能）。`launch play-reviews` は Play のカスタマーレビューを読み取り、返信を投稿します — Play Console を開くことなく。

**ビルドして届ける — iOS と Android**

- **プラットフォームごとに一つのコマンド。** `launch build ios` / `launch build android` は、prebuild → 署名 → サイズチェック → テスト用トラック（TestFlight / Play 内部）へのアップロードを実行します — EAS が走らせるのと同じフローです。
- **デフォルトで高速。** ccache は `pod install` 時に組み込まれ、DerivedData は温まったまま保たれ、ネイティブグラフのフィンガープリントが、ネイティブ依存関係が実際に変わったときだけクリーンビルドを強制します。JS の編集はインクリメンタルに再ビルドされます。`--clean` を付ければ最初からやり直します。
- **本物のダウンロードサイズチェック。** デバイスごとの実際のサイズ（App Thinning レポート／bundletool）を報告し、設定した `sizeBudgetMB` で判定します。
- **セーフティネット。** シミュレータビルド、`.app`、空の成果物のアップロードを拒否します。`--dry-run` は、ネットワーク・ビルド・アカウントへの変更を一切伴わずにパイプライン全体をリハーサルします。
- **意図的な一般公開リリース — ポータル不要。** テスト用トラックがデフォルトで、一般ストアへの公開は別個の、確認を伴う `launch release <platform>` です。iOS については App Store Connect API を端から端まで駆動します — バージョンの作成／再利用、輸出コンプライアンスへの回答、ビルドの添付、リリースノートの記述、即時／予約／段階的展開の選択、そして審査へのサブミット — そのためリリースにウェブサイトは決して必要ありません。`launch status [--watch] [--json]` は審査を追跡し（CI 終了コード付き）、`launch rollout pause|resume|complete` は段階的展開を操舵します。fastlane の役割はバイナリのアップロードのみに限定されています。
- **再ビルドせずに再署名。** `launch build:resign` は、保存済みの `.ipa`/`.aab` を異なる認証情報（新しいアカウントやプロファイル）で、成果物から直接再署名します — 再ビルドはありません。
- **完了通知。** `notify` ブロックは、ビルドやサブミットが終わったときに Slack/Discord の Webhook に通知を送り、かつ／またはシェルフックを実行します — 成功時 _も_ 失敗時 _も_ — そのため無人／CI 実行でも終わったことを知らせてくれます。EAS の `webhook` と同等で、ホスト型サービスは不要です。

**配信とアップデート**

- **内部配信。** `launch build <platform> --distribution internal` は、アドホックな iOS インストールリンク／Android `.apk` をあなた自身のバケットでホストします。テスターは `launch device add <udid>` で登録します。
- **OTA（無線越し）アップデート。** `launch update` は、`expo-updates` がすでに話している **Expo Updates プロトコル** 経由で JS／アセットのアップデートを公開します — コード署名され、あなた自身のバケット（S3 / R2 / Supabase）にホストされます。
- **不正なアップデートをロールバック。** `launch updates list/view` はチャンネルごとの履歴を表示します。`launch updates rollback` は不正な OTA を巻き戻します — 既知の良好なアップデートをプロモートするか、テスターを埋め込みバンドルへ戻します。

**テスター、チーム、レビュー、レポートを管理 — API キーのみで**

- **CLI から TestFlight を。** `launch testflight groups/create-group/testers/add/rm` は、同じ App Store Connect API キーでベータグループを管理し、テスターを招待します — ビルドアップロードを取り巻く管理レイヤーであり、Apple ID のパスワードも 2FA も不要です。
- **レビューの読み取りと返信。** `launch reviews [list]` はカスタマーレビューを読み取ります（レーティング／地域でフィルタ可能）。`launch reviews reply` は開発者の返信を投稿または差し替え、`launch reviews delete` はそれを削除します — App Store Connect を開くことなく。
- **販売、財務、分析レポート。** `launch reports sales/finance/analytics` は、App Store Connect の Sales & Trends、財務、分析の各レポート（gzip 圧縮された TSV）をスクリプト処理向けにあなたのマシンへ直接ダウンロードします — EAS が決して表に出さない数字です。
- **チームとアクセス。** `launch team list/invite/remove` は、同じ API キーで App Store Connect のチームメンバーと保留中の招待を読み取り、管理します — ロールを指定してメールで招待するか、取り消します。
- **サンドボックステスター。** `launch sandbox list/clear` は StoreKit のサンドボックステスターを一覧し、その購入履歴をクリアして、まっさらな状態からアプリ内課金を再テストできるようにします。

**検査とデバッグ**

- **ビルド履歴。** `launch builds list/view/log` はローカルの成果物インデックスを読み取ります — ID、デバイスごとのサイズ、成果物のパス、そして任意のビルドの生のビルドログです。
- **インストールして実行。** `launch run [id|latest]` は、ビルド済みの成果物を接続中のデバイスまたはシミュレータにインストールします（Android は `adb`/`bundletool`、iOS は `devicectl`）。
- **失敗を説明。** `launch diagnose` は `xcodebuild`/Gradle/CocoaPods のエラーを平易な言葉での原因と対処にマッピングします。`launch fingerprint` は、次のビルドがクリーンになるかインクリメンタルになるかの理由を示します。`--verbose` は進行スピナーの代わりに生のビルド出力をストリームします。

**オンボーディングと学習**

- **`launch demo`。** パイプライン全体をセットアップ不要でシミュレーションしたウォークスルーです（初回実行時に一度だけ自動再生されます）。どのコマンドにも付けられる `--explain` と `launch explain <topic>` が、用語を必要に応じて解説します。
- **静かな自己アップグレード。** より新しい npm リリースを取得して、その上であなたのコマンドを再実行します — 一日一回に絞られ、CI 内、パイプ接続時、エージェントに対しては何もしません。

## Launch 対 EAS

Launch は、`eas build` → `eas submit` → `eas update` の同じパイプライン（加えて `eas metadata` と `eas credentials`）を、あなた自身のハードウェア上で走らせます — そして EAS があなたに委ねているストア設定のステップもカバーします。同じワークフロー上で両者が異なる点は以下のとおりです。

| Expo EAS では                                                                                                      | Launch では                                                                                                                                             |
| ------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ビルドのコンピュートは **Expo のクラウド** で実行され、**月額 $19〜$199** ＋ビルドごとの料金                       | **あなた自身のマシン** でビルド — **コンピュート費用 $0**、MIT ライセンス、ビルド無制限                                                                 |
| ビルドは共有クラウドで **キュー待ち** になり、ときには何時間も                                                     | ビルドはあなたのハードウェア上で **即座に開始** — キューなし                                                                                            |
| 無料枠のビルドは **45 分のタイムアウトで制限**                                                                     | **タイムアウトなし** — ビルドは必要なだけ走ります                                                                                                       |
| Apple ID の **2FA** プロンプト／期限切れのコードがビルドを中断                                                     | **App Store Connect API キー（JWT）** で認証 — パスワードも 2FA も不要                                                                                  |
| ツールチェーン／Node は **ビルドイメージにピン留め**。ローカルの `.env` は解決されない                             | あなた自身の Xcode/Node/ツールチェーンと、文書化された **環境優先順位のはしご**（`--print-env` で監査）                                                 |
| **アプリ内課金とサブスクリプション** は ASC の UI での手作業                                                       | `launch sync` が **IAP、サブスクリプション、ケイパビリティ** を `launch.config.ts` から整合                                                             |
| EAS は **毎ビルドでバンドル ID のケイパビリティを書き換える**（トグルを潰す）                                      | `launch sync` は **最小限の安全な差分** を適用 — 管理対象外のケイパビリティはそのまま                                                                   |
| `eas metadata` は **iOS のみ**                                                                                     | `launch metadata` は **iOS _と_ Android** の掲載情報を同期                                                                                              |
| Play の **IAP、サブスクリプション、トラック、レビュー** は Play Console の UI 行き                                 | `launch play-products` / `play-subscriptions` が **同じ** 設定から Play を整合。`launch play-tracks` / `play-reviews` が API 経由でトラックと返信を駆動 |
| **Game Center、Wallet、App Clips、アプリ内イベント、A/B 実験、地域、アクセシビリティ** は **ポータルのみ**         | それぞれが **コードとしての設定** — `launch game-center` / `wallet` / `app-clips` / `events` / `experiments` / `availability` / `accessibility`         |
| `eas submit` の後、**App Store のリリース**（バージョン、コンプライアンス、ノート、展開）は **ポータルでの手作業** | `launch release` が **API** 経由で駆動 — その後 `launch status --watch` と `launch rollout` が追跡・操舵                                                |
| **レビュー、レポート、TestFlight 管理** はウェブサイトへの往復を意味する                                           | `launch reviews` / `launch reports` / `launch testflight` が **API キー** 経由で実行 — ポータル不要                                                     |
| **EAS Update** は OTA アップデートを **Expo のサーバー** にホスト（有料）                                          | `launch update` は **同じ Expo Updates プロトコル** を **あなた自身のバケット**（S3/R2/Supabase）から配信                                               |
| **内部配信** ビルドは Expo がホスト                                                                                | `--distribution internal` がアドホックな `.ipa`/`.apk` を **あなた自身のバケット** にホスト。`launch device`                                            |
| 署名 **認証情報は Expo のサーバーに置かれることがある**                                                            | 鍵は **あなたの OS キーチェーン** に留まる — マシンを離れるのは **CSR** だけ                                                                            |
| ビルドの **成果物は Expo にホストされる**                                                                          | 成果物は **あなた自身のストレージ**（ローカル、または S3 / R2 / Supabase）に着地                                                                        |
| **Mac がない？** EAS の有料クラウドが唯一の道                                                                      | **Mac がない？** **あなた自身の AWS** 内のクラウド Mac、**SSH** 越しの任意の Mac、または **`eas build`** への引き継ぎ                                   |
| **クローズドな SaaS** — プロプライエタリでベンダーロックイン                                                       | **MIT、オープンソース** — `fastlane`/Gradle/プラットフォーム API、差し替え可能なプロバイダー、移行すべきものは何もなし                                  |

## 必要要件

- **iOS:** **Apple Developer Program** への登録（年額 99 ドル） — [こちらで登録](https://developer.apple.com/programs/enroll/) — その後、**Xcode** ＋コマンドラインツールが入った macOS、**fastlane**（`brew install fastlane`）、そして **App Store Connect API キー**（`.p8` ＋ Key ID ＋ Issuer ID） — [こちらで生成](https://appstoreconnect.apple.com/access/integrations/api)。Mac がない？ [Mac なしでビルドする](#mac-なしでビルドする) を参照してください。
- **Android:** **Google Play デベロッパー アカウント**（買い切り 25 ドル） — [こちらで登録](https://play.google.com/console/signup) — その後、**JDK**（任意の OS — Mac は不要）と **Google Play サービスアカウント** の JSON キー。
- すべてのプラットフォームで **Node 20+**。

`launch doctor` をいつでも実行すれば、上記すべてを確認できます。

## クイックスタート

ペイウォールからテスト用トラックまで、5 つのコマンドで（Google Play 向けには `ios` → `android` に置き換えてください）。

```bash
launch init                 # scaffold launch.config.ts + .env.example, tailored to your repo
launch creds set-key        # import your store API key into the OS keychain
launch creds setup          # register the app id + create/reuse signing assets
launch build ios --dry-run  # rehearse the whole flow — no network, no build, no account changes
launch build ios            # build, sign, size-check, and upload to the testing track
```

`launch build` はキャッシュされた認証情報を黙って再利用します。それらが見つからない場合は、その場でプロビジョニングを申し出ます。一般公開リリースは別個の、意図的な `launch release <platform>` です。

アプリ内課金やサブスクリプションを販売しますか？ それらを `launch.config.ts` に宣言し、`launch sync` を実行すれば、App Store Connect 上で作成・整合されます — ポータルをクリックして回る必要はありません。

すでにアプリを提供中ですか？ `launch adopt` は、ライブの App Store Connect セットアップ — 商品、ケイパビリティ、署名、掲載情報 — を読み取り、それをワンステップで設定に書き戻します。これにより `sync` で前進させていけます。

## 設定

アプリの事実情報（バンドル ID、バージョン）は各アプリの Expo 設定 — `app.json` または `app.config.{ts,js}` — から読み取られるため、決して重複しません。`launch.config.ts` には Launch 固有の設定のみを保持します。

```ts
import { defineConfig } from "launch-store";

export default defineConfig({
  // appRoots: ["./apps"],   // for a monorepo; omit to scan the repo root
  credentials: "local", // OS keychain + ~/.launch
  storage: "local", // ~/.launch/artifacts (swap for s3/r2/supabase later)
  buildEngine: "fastlane", // "fastlane" (local Mac) · "remote-mac" (AWS EC2 Mac) · "eas" (Expo cloud)
  // submit: "app-store-connect", // or "eas" to submit through Expo

  // Only needed to build iOS without a Mac via `--remote aws` — see "Building without a Mac".
  // aws: { region: "us-east-1" },

  profiles: {
    // `env` is inline per-profile vars; `envFile` renames the base dotenv. Precedence, highest first:
    // --env flags › keychain secrets › profile `env:` › .env.local (--include-local) › .env.<profile> › .env
    production: { name: "production", envFile: ".env", env: {}, sizeBudgetMB: 200 },
  },

  // Ping a Slack/Discord webhook and/or run a shell hook when a build or submit finishes (success or
  // failure). Both fields are optional; omit `notify` entirely for no notifications.
  // notify: { webhookUrl: "https://hooks.slack.com/services/…", command: "say build done" },

  // In-app purchases & subscriptions, keyed by bundle id — `launch sync` reconciles these onto App Store
  // Connect (and `launch play-products` / `play-subscriptions` onto Google Play). Capabilities aren't
  // declared here; they're read from app.json's `ios.entitlements`. Omit if your app sells nothing.
  // products: { "com.company.app": { subscriptionGroups: [/* … */], inAppPurchases: [/* … */] } },

  // Launch-native App Store Connect sections — each reconciled by its own command, declared inline here
  // (or, for back-compat, as a standalone `*.config.json` sidecar). Per-app ones are keyed by iOS bundle
  // id; Wallet & EU distribution are team-level. See examples/hello-world for a worked copy of each.
  // gameCenter: { "com.company.app": { achievements: [/* … */], leaderboards: [/* … */] } },
  // appClips: { "com.company.app": { clips: { "com.company.app.Clip": { action: "OPEN" } } } },
  // releaseAttributes: { "com.company.app": { pricing: { customerPrice: 9.99 }, categories: { primary: "PRODUCTIVITY" } } },
  // wallet: { merchantIds: [/* … */], passTypeIds: [/* … */] },
  // euDistribution: { domains: [/* … */] },
});
```

`launch build <platform> --print-env` を実行すると、完全に解決された環境と、各値がどこから来たか（シークレットはマスク表示）が、ビルドが一つでも走る前に確認できます。フル機能の、デュアルプラットフォーム（iOS + Android）の例 — 商品、オファー、リリース属性から Game Center、Wallet、EU 配信、Google Play カタログまで、コードとしての設定のあらゆる面を行使するもの — が [`examples/hello-world`](./examples/hello-world) にあります（機能ごとのツアーはその README を参照してください）。

## Mac なしでビルドする

iOS の署名は macOS 専用のため、Windows/Linux の開発者はどこかに Mac が必要です。`launch`（ウィザード）を実行するか、直接いずれかの道を選んでください。Android は JDK が動く場所ならどこでもビルドできるため、これは Android には当てはまりません。

| 道                    | 何が起きるか                                                                                                                    | コスト                                                                                                               |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| **AWS クラウド Mac**  | Launch が **あなた自身の** AWS アカウントに EC2 Mac をプロビジョニングし、ビルド＋署名＋サブミットし、その後取り壊します。      | AWS に直接支払い — **24 時間セッションあたり最低 〜$16**（Apple のライセンスが 24 時間の下限を厳格に定めています）。 |
| **Mac に接続（SSH）** | 到達できる任意の Mac でビルド — 同僚の Mac、MacStadium、手動で起動したインスタンスなど。                                        | その Mac にかかる費用は何であれ。                                                                                    |
| **Expo EAS**          | Launch が `eas-cli` を Expo のクラウド上で端から端まで（`eas build` → ダウンロード → `eas submit`）オーケストレーションします。 | Expo の **無料枠**、月次の上限付き。                                                                                 |

```bash
launch build ios --remote aws            # build on a cloud Mac in your AWS account
launch build ios --remote ec2-user@host  # build on a Mac you reach over SSH
launch cloud doctor                      # check AWS creds, region, Mac-host quota, IAM
launch cloud status                      # live host: age, cost so far, releasable-after time
launch cloud teardown                    # stop + release the host (warns about the 24h floor)
```

リモートビルドは、あなたの署名鍵の一時的なコピーを、明示的な同意のもとで **あなた自身の** ホストにアップロードし、その後シュレッド（完全消去）します — 決して他人のサーバーへは送りません。たまにしか iOS をビルドしない場合、GitHub Actions の macOS ランナーは EC2 Mac より安価です。ここでの Launch の価値は、どこでも同じ鍵を使って、あなた自身のアカウント内で自動化できる点にあります。

## 認証情報の扱われ方

- API キー（`.p8`）、配布用の秘密鍵、Android アップロードキーは、あなたの **OS キーチェーン** に置かれます。
- iOS の証明書は、`~/.launch/credentials/` 以下にパスワード保護された `.p12`（chmod 600）としてもバックアップされます。そのパスワードはキーチェーンに格納され、ファイルのそばには決して置かれません。
- あなたの秘密鍵はローカルで生成されます — Apple に送られるのは CSR だけです。
- Launch は新しい証明書を作成せず、既存の配布証明書を再利用します（ストアは証明書数に上限を設けています）。

## FAQ

**Launch とは何ですか？** Launch は、オープンソースのセルフホスト型 Expo EAS 代替ツールです。あなた自身のマシン上で、あなた自身の鍵を使い、ビルドごとの課金なしに、Expo / React Native アプリを TestFlight と Google Play へビルド・署名・出荷します。EAS が行う build → submit → update と同じパイプラインを実行し、さらに EAS が App Store Connect と Play Console のウェブサイトに委ねているストア設定ステップ — アプリ内課金、サブスクリプション、ケイパビリティ、掲載メタデータ — をコードとして追加します。

**Launch は Expo EAS の無料オープンソース代替ですか？** はい。Launch は MIT ライセンスで完全にオープンソースであり、ビルドはあなた自身のハードウェア上で実行されるため、ビルドごとの料金も、ビルド分単位のメーターも、月額サブスクリプションも一切ありません — EAS の有料ティアである月額 $19–$199 とビルドごとの超過料金とは対照的です。唯一の任意コストは、Mac なしで iOS をビルドする必要がある場合にクラウド Mac をレンタルすることです。

**Launch は Expo EAS とどう違いますか？** EAS はあなたのビルドを Expo のクラウドで実行し、認証情報・成果物・OTA アップデートを Expo のサーバーに保持します。Launch は同一のパイプラインをあなた自身のマシン上で実行し、署名鍵を OS キーチェーンに保持し、成果物と OTA アップデートをあなた自身のバケット（S3 / R2 / Supabase）に保存します — そして EAS が扱わないストア設定（IAP、サブスクリプション、ケイパビリティ、iOS と Android の掲載情報）をコードとして管理します。コマンドは一対一で対応します: `eas build` → `launch build`、`eas submit` → `launch release`、`eas update` → `launch update`、`eas metadata` → `launch metadata`、`eas credentials` → `launch creds`。

**Mac なしで iOS アプリをビルドできますか？** iOS のコード署名とビルドツールチェーンは macOS 専用のため、どこかに Mac が必要です — ただしそれはあなたの Mac である必要はありません。Launch は、あなた自身の AWS アカウント内のクラウド Mac（EC2 Mac）をプロビジョニングしたり、SSH で到達できる任意の Mac でビルドしたり、Expo EAS のクラウドに処理を引き継いだりできます。Android は JDK が動く場所ならどこでもビルドでき、Mac は一切不要です。

**Launch は Android と Google Play に対応していますか？** はい。Launch は Android アプリをビルド・署名して Google Play にアップロードし、App Store Connect を駆動するのと同じ `launch.config.ts` カタログから Play のアプリ内商品、サブスクリプション（基本プラン＋オファー）、リリーストラック、レビュー返信を整合させます — 両ストアに対する唯一の信頼できる情報源です。

**Launch は EAS Update のような OTA アップデートに対応していますか？** はい。`launch update` は、あなたの `expo-updates` ランタイムがすでに話している Expo Updates プロトコル経由で JS とアセットのアップデートを公開します — コード署名され、Expo のサーバーではなくあなた自身のバケット（S3 / R2 / Supabase）にホストされます。`launch updates rollback` は、既知の良好なアップデートをプロモートするかクライアントを埋め込みバンドルへ戻すことで、問題のあるリリースを巻き戻します。

**Expo EAS から Launch へ移行するにはどうすればいいですか？** コマンドを一対一で入れ替えてください（`eas build` → `launch build`、`eas submit` → `launch release`、`eas update` → `launch update`、`eas credentials` → `launch creds`、`eas metadata` → `launch metadata`）。すでにアプリを提供中なら、`launch adopt` がライブの App Store Connect セットアップ — 商品、ケイパビリティ、署名、掲載情報 — を読み取り、ワンステップで `launch.config.ts` に書き戻します。また Mac がない場合は引き続き `eas build` に引き継げるため、段階的に移行できます。

**Launch は単なる App Store Connect SDK や MCP ラッパーですか？** いいえ。App Store Connect の SDK や MCP サーバーは Apple API の一部分をラップするにすぎません。Launch は Apple と Google にまたがるリリース全体を駆動します — コード署名、ネイティブビルド、サイズチェック、コードとしてのストア設定、確認済みの一般公開リリース、そして OTA アップデート — これらはどれも API ラッパーが触れない領域です。API クライアントではなくセルフホスト型の Expo EAS が欲しいなら、それが Launch です。

**Launch は Fastlane とどう違いますか？** Fastlane はビルディングブロックであり、Launch はそれをオーケストレーションします。Launch はバイナリアップロードのステップにのみ fastlane を使用し、その周囲にリリース全体を構築します: 認証情報プロビジョニング、ビルド、実際のダウンロードサイズチェック、両ストア向けのコードとしてのストア設定、意図的な一般公開リリース、段階的ロールアウト制御、そして OTA アップデート — すべて一つの型付き `launch.config.ts` から。

**署名鍵とシークレットはどこに保存されますか？** あなたの OS キーチェーンに保存されます。App Store Connect API キー（`.p8`）、配布用の秘密鍵、Android アップロードキーは、リポジトリにも誰かのサーバーにも決して触れません — Apple に送られるのは証明書署名要求（CSR）だけです。ビルドシークレットも `launch secret` 経由でキーチェーンに保存されるため、コミットされた `.env` の外に留まります。

**Launch を実行するには何が必要ですか？** すべての環境で Node 20+。iOS の場合: Xcode とコマンドラインツールが入った macOS、fastlane、および App Store Connect API キー（`.p8` ＋ Key ID ＋ Issuer ID） — Mac がない場合はリモート Mac。Android の場合: JDK（任意の OS）と Google Play サービスアカウントの JSON キー。`launch doctor` を実行すれば一度にすべて確認できます。

**Launch の料金はいくらですか？** Launch 自体は無料（MIT）です。支払うのは、いずれにせよ支払うものだけです: あなた自身のビルドハードウェア（またはローカル Mac なしで iOS をビルドする場合のクラウド Mac 時間）、そして通常の Apple Developer（$99/yr）と Google Play（一回限り $25）の登録料。ビルドごとの料金もサブスクリプションも一切ありません。

**Launch は CI で動きますか？** はい。`launch ci init` はホスト型 macOS ランナー上の GitHub Actions ワークフローをスキャフォールドし、すべてのコマンドは CI を検出したとき、stdout がパイプ接続されたとき、またはエージェントに対して、非対話モードに退避します — そのため同じフローが無人で実行できます。

**Launch はどのフレームワークに対応していますか？** Expo 設定（`app.json` / `app.config.{ts,js}`）と `expo prebuild` を通じて自身を記述する Expo およびベア React Native アプリ。Launch はバンドル ID、バージョン、エンタイトルメントをそこから読み取るため、何も重複しません。

**Launch はストアメタデータ、アプリ内課金、サブスクリプションを管理できますか？** はい — 両ストアに対してコードとして管理します。`launch sync` は IAP、サブスクリプション、価格、ケイパビリティ、ロケールごとの掲載情報（テキスト、スクリーンショット、プレビュー）を App Store Connect に整合させます。`launch metadata` は iOS と Android の掲載情報をカバーします。`launch play-products` / `launch play-subscriptions` は Google Play カタログを駆動します。各コマンドは読み取り専用のプラン → 確認 → 適用で実行されるため、ライブ版や審査中のバージョンを上書きすることはありません。

**Launch はホスト型サービスにロックインされますか？** いいえ — ホスト型のものも独自仕様のものも何もありません。Launch は MIT ライセンスで、fastlane、Gradle、各プラットフォーム自身の API の上に構築されており、ストレージ／認証情報／ビルド／サブミットのプロバイダーを差し替え可能です。鍵、成果物、アップデートはあなたが管理するインフラに置かれるため、あとで移行しなければならないものは何もありません。

**始めるにはどうすればいいですか？** `npm install --global launch-store`（またはプロジェクトごとに `--save-dev`）でインストールし、`launch demo` を実行すると 60 秒のシミュレーションウォークスルーを体験できます — セットアップもアカウントも不要です。準備ができたら: `launch init` → `launch creds set-key` → `launch creds setup` → `launch build ios`。
